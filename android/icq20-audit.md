## 2. Безопасность

### 🔴 Критические уязвимости

| # | Проблема |Severity | Описание |
|---|----------|---------|----------|
| **C1** | gRPC без TLS | **HIGH** | gRPC порт 50051 слушает `0.0.0.0` без шифрования. Любой может подключиться к другой ноде через MITM, подделать `RelayMessage`, `SyncUser`, `DrainOffline`. |
| **C2** | GRPC auth HMAC в plaintext | **HIGH** | HMAC-SHA256 для гRPC подписан, но канал открыт. Replay attack возможен если кто-то перехватывает трафик. |
| **C3** | JWT secret в .env (plaintext) | **MEDIUM** | Если кто-то получит доступ к файлу — все токены скомпрометированы. |
| **C4** | Нет rate limiting на REST endpoints | **MEDIUM** | `/api/register` ограничен, но `/api/create-group`, `/api/contacts/discover`, `/api/group/id/members` не защищены от флуда. |
| **C5** | Offline group messages без auth | **LOW** | Групповые сообщения сохраняются в БД без шифрования на сервере. |

### 🟡 Предупреждения

| # | Проблема | Описание |
|---|----------|----------|
| W1 | PIN передаётся по сети в POST | Не хешированный, TLS защищает но при компрометации TLS — PIN открыт |
| W2 | Нет 2FA | Только PIN (6 цифр) для входа |
| W3 | Ключи не хранятся в Keychain (iOS) | UserDefaults (plaintext) — критично для App Store |
| W4 | Формат E2E совместим? | Android + iOS должны использовать одинаковый DER/SPKI формат |
| W5 | CORS: `Access-Control-Allow-Origin *` | На сайте (через Caddy) — разрешает любые домены XSS |
| W6 | CORS wildcard на корп-ноде | `*` — любой сайт может делать API-запросы |

### ✅ Безопасные решения

| # | Реализация |
|---|------------|
| S1 | E2E шифрование: P-256 ECDH + HKDF-SHA256 + AES-256-GCM |
| S2 | Сервер не видит plaintext сообщений |
| S3 | PIN хеширован Argon2id на сервере (лучший KDF 2024) |
| S4 | WS sender_uin перезаписывается из JWT |
| S5 | Rate limiting: 20 msg/10s на WS |
| S6 | Login lockout: 5 fails → 15min бан |
| S7 | JWT с audience/issuer validation |
| S8 | Обфускация трафика (DPI evasion) |
| S9 | `delete_account` — GDPR удаление |

## 3. Отказоустойчивость

### Слабые места

| # | Проблема | Severity | Fix |
|---|----------|----------|-----|
| R1 | Нет репликации БД | **HIGH** | Один PostgreSQL — если упадёт, все данные потеряны. Нужен streaming replication или pgBouncer. |
| R2 | Нет auto-restart при DB fail | **HIGH** | Если PostgreSQL упадёт, сервер продолжает работать с `db_healthy = false` но не перезапускается. |
| R3 | Нет бэкапов БД автоматически | **MEDIUM** | Бэкапы есть (10 файлов в `/opt/icq20/backups/`) но cronjob не настроен автоматически. |
| R4 | Нет health check для Caddy → Backend | **LOW** | Caddy не знает если backend упал. |
| R5 | Оффлайн-очередь макс 500, нет TTL для групп | **LOW** | Групповые сообщения не чистятся автоматически. |
| R6 | WS timeout 90s, ping 30s | **OK** | Адекватно для мобильных сетей. |
| R7 | Disk 54% использовано | **Medium** | 6.7GB свободно на 15GB диске. |

### ✅ Хорошие решения

| # | Реализация |
|---|------------|
| G1 | PostgreSQL health check каждые 30s |
| G2 | Offline TTL 30 дней (автоочистка) |
| G3 | gRPC peer auto-reconnect каждые 15s |
| G4 | Heartbeat каждые 30s |
| G5 | Hash ring routing для нод |
| G6 | systemd restart=always |

### Что готово ✅

## 6. Оценка готовности к продакшену

### По модулям:

| Модуль | Готовность | Что не хватает |
|--------|------------|----------------|
| **Сервер Rust** | 85% | gRPC TLS, REST rate limit, мониторинг |
| **E2E Crypto** | 80% | Key verification (Signal-style), iOS совместимость |
| **Auth (JWT + PIN)** | 90% | 2FA, token rotation |
| **WebSocket** | 85% | Group msg routing (нужно тестирование) |
| **Offline + Push** | 50% | APNs не настроен, FCM опционально |
| **Сайт** | 95% | Все фичи есть, minor UI fixes |
| **Android** | 95% | Полностью рабочий |
| **iOS** | 35% | Нет сборки, нет Keychain, xcodeproj |
| **Биржа** | 40% | Серверный pay endpoint без JWT — не готов, ЮKassa не настроена |
| **Мониторинг** | 20% | Нет метрик, нет алертов, нет Grafana |
| **Бэкапы** | 40% | Вручную есть, cron нет, restore не тестирован |
| **Документация** | 75% | API docs есть, deploy guide частично |

### Общая оценка: **~55% к коммерческому запуску**

## 7. Roadmap до продакшена (приоритет)

### 🔴 Обязательно (1-2 недели)
1. **gRPC TLS** — шифровать меж-нодовой трафик (2 часа)
2. **REST rate limiting** — все endpoints (1 день)
3. **CORS: убрать wildcard** — `iromashka.ru` (30 мин)
4. **Keychain на iOS** — вместо UserDefaults (2 часа)
5. **Бэкапы по cron** — ежедневно, хранить 30 дней (1 день)
6. **Мониторинг** — UptimeRobot + алёрты в Telegram (1 день)

### 🟡 Крайне рекомендуется (2-4 недели)

## 8. Инфраструктурные рекомендации

| Сейчас | Рекомендация |
|--------|-------------|
| 1 сервер 15GB | Добавить DB-сервер (отдельный), 50GB+ RAM |
| Без мониторинга | UptimeRobot + Grafana + Prometheus |
| Бэкап раз в 6ч (ручной) | Автоматически + тестирование restore |
| 1 ноду | Минимум 2 ноды (primary + backup) |
| Без CDN | Не нужно (мессенджер, не контент) |
| Caddy один | Caddy + reverse proxy + WAF (CloudFlare) |

---

**Итого:** Серверная часть стабильна и функциональна. Главное, что блокирует релиз — **безопасность (gRPC TLS, REST rate limit), отсутствие iOS сборки, отсутствие мониторинга**. Без этих вещей нельзя запускать публично.

Для внутреннего использования (корп-мессенджер на 10-100 человек) — **можно запускать сейчас** с minor фиксами (gRPC TLS + iOS Keychain).
